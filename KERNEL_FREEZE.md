## Regla de madurez (carga de la prueba invertida)

> **El diseño del kernel se considera congelado hasta que exista evidencia suficiente para justificar un cambio metodológico.**

| Antes | Ahora |
| ----- | ----- |
| Añadimos una mejora porque parece buena | **No** cambiamos el kernel salvo que un **META-E** muestre que merece cambiar |

---

## Filtro ante una idea nueva

| Pregunta | Acción |
| -------- | ------ |
| ¿Es **ingeniería**? | Solo si respeta **conservación epistemológica** (no reabre el diseño del método) |
| ¿Es **metodológica**? | Se formula como **hipótesis META-E** (protocolo, E-M*, puede morir) |
| ¿No cabe en ninguna? | **Aún no** está definida para entrar al kernel |

Eso evita crecimiento innecesario.

## Qué se hace y qué no

| Hacer | No hacer |
| ----- | -------- |
| Ejecutar / formalizar META-E (E001…) | Inventar componentes “porque sí” |
| Acumular perfiles E-M1…E-M5 | Sobreteorizar el lab |
| Ingeniería que conserve evidencia | Mezclar refactor con “mejora metodológica” no medida |

## Señal de cambio de fase

El logro no es un componente concreto (Core, Auditor, Selector).

Es haber convertido casi todas las decisiones importantes en **hipótesis investigables**:

- matemáticas,
- metodológicas,
- arquitectónicas,
- mejoras.

El proyecto deja de crecer por **acumulación de ideas** y empieza a crecer por **acumulación de evidencia**.

Esa es la señal de paso de **diseño** a **investigación**.
